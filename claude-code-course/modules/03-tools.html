工具的"基因蓝图"：Tool 类型

让我们看看 src/Tool.ts 中工具的类型定义——这是所有工具必须遵守的"契约"。

src/Tool.ts (第 362-410 行,精简版) 
export type Tool<
  Input extends AnyObject = AnyObject,
  Output = unknown,
  P extends ToolProgressData = ToolProgressData,
> = {
  // 工具的唯一名称
  readonly name: string

  // 可选的别名(用于重命名后的兼容性)
  aliases?: string[]

  // 输入参数的 schema (Zod 类型)
  readonly inputSchema: Input

  // 输出类型的 schema
  outputSchema?: z.ZodType<unknown>

  // ===== 核心方法 =====

  // 执行工具的主函数
  call(
    args: z.infer<Input>,
    context: ToolUseContext,
    canUseTool: CanUseToolFn,
    parentMessage: AssistantMessage,
    onProgress?: ToolCallProgress<P>,
  ): Promise<ToolResult<Output>>

  // 生成工具描述(给 AI 看的)
  description(
    input: z.infer<Input>,
    options: { ... }
  ): Promise<string>

  // ===== 安全和权限 =====

  // 验证输入是否合法
  validateInput?(
    input: z.infer<Input>,
    context: ToolUseContext,
  ): Promise<ValidationResult>

  // 检查权限(核心安全机制)
  checkPermissions(
    input: z.infer<Input>,
    context: ToolUseContext,
  ): Promise<PermissionResult>

  // ===== 元数据方法 =====

  // 是否启用这个工具
  isEnabled(): boolean

  // 是否是只读操作
  isReadOnly(input: z.infer<Input>): boolean

  // 是否是破坏性操作(删除、覆盖等)
  isDestructive?(input: z.infer<Input>): boolean

  // 是否可以并发执行
  isConcurrencySafe(input: z.infer<Input>): boolean

  // ===== UI 渲染方法 =====

  // 渲染工具使用消息(在终端中显示)
  renderToolUseMessage(
    input: Partial<z.infer<Input>>,
    options: { theme: ThemeName; verbose: boolean }
  ): React.ReactNode

  // 渲染工具结果消息
  renderToolResultMessage?(
    content: Output,
    progressMessages: ProgressMessage<P>[],
    options: { ... }
  ): React.ReactNode

  // 用户可见的工具名称
  userFacingName(input: Partial<z.infer<Input>> | undefined): string
}

这个类型定义在说什么？

1-5

Tool 是一个泛型类型，可以指定输入类型、输出类型、进度类型。

19-26

call() 是工具的灵魂——这里是实际逻辑执行的地方。它接收参数、上下文、权限检查函数、进度回调，返回结果。

34-43

双重安全机制：validateInput 检查输入合法性(路径是否存在、参数是否有效)，checkPermissions 检查用户是否允许(核心安全)。

48-57

元数据方法帮助系统理解工具的性质：是否只读？是否破坏性？是否可并发？这些信息用于优化执行和保护用户。

61-73

UI 渲染方法：工具不仅执行操作，还要在终端中漂亮地展示自己在做什么。这就是为什么你能看到彩色、格式化的输出。

🎯 设计模式：接口契约

Tool 类型定义了一个接口契约——所有工具必须实现这些方法。这让系统可以统一处理 40+ 个工具，而不需要为每个工具编写特殊逻辑。这是多态的经典应用。

实战案例：BashTool 如何工作

让我们看看最常用的工具之一——BashTool。它负责执行所有终端命令。

src/tools/BashTool/BashTool.tsx (第 95-172 行,精简版) 
// 检查 bash 命令是否是搜索或读取操作
export function isSearchOrReadBashCommand(command: string): {
  isSearch: boolean;
  isRead: boolean;
  isList: boolean;
} {
  // 尝试解析命令
  let partsWithOperators: string[];
  try {
    partsWithOperators = splitCommandWithOperators(command);
  } catch {
    // 如果解析失败(语法错误),不是搜索/读取命令
    return { isSearch: false, isRead: false, isList: false };
  }

  if (partsWithOperators.length === 0) {
    return { isSearch: false, isRead: false, isList: false };
  }

  let hasSearch = false;
  let hasRead = false;
  let hasList = false;
  let hasNonNeutralCommand = false;

  // 遍历命令的每个部分
  for (const part of partsWithOperators) {
    // 跳过重定向操作符
    if (part === '>' || part === '>>' || part === '>&') {
      continue;
    }
    // 跳过管道和逻辑操作符
    if (part === '||' || part === '&&' || part === '|' || part === ';') {
      continue;
    }

    const baseCommand = part.trim().split(/\s+/)[0];
    if (!baseCommand) continue;

    // 跳过语义中性的命令(echo, printf 等)
    if (BASH_SEMANTIC_NEUTRAL_COMMANDS.has(baseCommand)) {
      continue;
    }

    hasNonNeutralCommand = true;

    // 检查是否是搜索/读取/列表命令
    const isPartSearch = BASH_SEARCH_COMMANDS.has(baseCommand);
    const isPartRead = BASH_READ_COMMANDS.has(baseCommand);
    const isPartList = BASH_LIST_COMMANDS.has(baseCommand);

    // 如果有任何部分不是搜索/读取/列表,整个命令就不可折叠
    if (!isPartSearch && !isPartRead && !isPartList) {
      return { isSearch: false, isRead: false, isList: false };
    }

    if (isPartSearch) hasSearch = true;
    if (isPartRead) hasRead = true;
    if (isPartList) hasList = true;
  }

  // 如果只有中性命令,不可折叠
  if (!hasNonNeutralCommand) {
    return { isSearch: false, isRead: false, isList: false };
  }

  return { isSearch: hasSearch, isRead: hasRead, isList: hasList };
}

这段代码在做什么？

1-6

定义函数：判断一个 bash 命令是否是"搜索"或"读取"操作。这用于 UI——这类命令的输出可以折叠显示。

8-13

安全解析：尝试解析命令。如果命令语法错误，直接返回 false(不是搜索/读取命令)。

25-42

遍历命令的每个部分。跳过操作符(管道、重定向)和"语义中性"的命令(echo, printf——它们不影响命令的读/写性质)。

44-53

检查每个命令是否在预定义的搜索/读取/列表命令集合中。关键规则：所有部分都必须是这类命令，整个命令才算可折叠。

全局逻辑

为什么需要这个？UI 优化。cat file.txt 的输出可以折叠，但 rm -rf / 绝对不能折叠——它是破坏性操作，必须显眼！

BashTool 还定义了几个重要的命令集合：

// 搜索命令 const BASH_SEARCH_COMMANDS = new Set([ 'find', 'grep', 'rg', 'ag', 'ack', 'locate', 'which', 'whereis' ]); // 读取/查看命令 const BASH_READ_COMMANDS = new Set([ 'cat', 'head', 'tail', 'less', 'more', 'wc', 'stat', 'file', 'strings', 'jq', 'awk', 'cut', 'sort', 'uniq', 'tr' ]); // 列表命令 const BASH_LIST_COMMANDS = new Set([ 'ls', 'tree', 'du' ]);

这些集合帮助系统理解命令的语义——不仅仅执行，还要理解它在做什么。

权限系统：保护用户的最后防线

Claude Code 的权限系统是它最重要的安全机制。让我们看看它如何工作。

权限检查的三个层级

总是允许

Always Allow

自动批准的操作，不需要询问

  • 读取项目目录内的文件
  • 执行 git status, git diff
  • 列出目录 ls
总是询问

Always Ask

每次都需要用户确认

  • 修改文件
  • 执行 git push, git commit
  • 运行可能有副作用的命令
总是拒绝

Always Deny

自动拒绝，不允许执行

  • 访问敏感目录(如 ~/.ssh)
  • 执行危险命令(如 rm -rf /)
  • 用户明确禁止的操作

🎯 为什么需要这么复杂的权限系统？

因为 AI 可能会犯错。它可能误解你的意图，生成错误的命令，或者被提示注入攻击利用。权限系统确保即使 AI 想做坏事，它也做不了——除非你明确同意。

权限规则的代码实现

让我们看看 src/utils/permissions/permissions.ts 中权限检查的实际代码。

src/utils/permissions/permissions.ts (第 1-100 行,精简版) 
import type { ToolPermissionContext, ToolUseContext } from '../../Tool.js'
import type { PermissionResult } from './PermissionResult.js'

// 权限检查的主入口
export async function checkPermissions(
  toolName: string,
  input: Record<string, unknown>,
  context: ToolUseContext
): Promise<PermissionResult> {
  const { mode, alwaysAllowRules, alwaysDenyRules, alwaysAskRules } =
    context.permissionContext

  // 第 1 步：检查「总是拒绝」规则
  const denyRule = getDenyRuleForTool(context.permissionContext, {
    name: toolName,
    ...input
  })

  if (denyRule) {
    return {
      behavior: 'deny',
      reason: 'rule',
      message: `操作被规则拒绝: ${denyRule}`
    }
  }

  // 第 2 步：检查「总是允许」规则
  const allowRule = getAllowRuleForTool(context.permissionContext, {
    name: toolName,
    ...input
  })

  if (allowRule) {
    return {
      behavior: 'allow',
      updatedInput: input
    }
  }

  // 第 3 步：检查「总是询问」规则
  const askRule = getAskRuleForTool(context.permissionContext, {
    name: toolName,
    ...input
  })

  if (askRule) {
    return {
      behavior: 'ask',
      message: '需要用户确认此操作'
    }
  }

  // 第 4 步：根据权限模式决定默认行为
  if (mode === 'auto') {
    // 自动模式：大多数操作自动允许
    return { behavior: 'allow', updatedInput: input }
  } else if (mode === 'plan') {
    // 计划模式：询问用户
    return { behavior: 'ask' }
  } else {
    // 默认模式：对破坏性操作询问
    const isDestructive = await isDestructiveOperation(toolName, input)
    return isDestructive
      ? { behavior: 'ask' }
      : { behavior: 'allow', updatedInput: input }
  }
}

// 检查操作是否是破坏性的
async function isDestructiveOperation(
  toolName: string,
  input: Record<string, unknown>
): Promise<boolean> {
  // 文件删除、覆盖、git push 等都是破坏性操作
  if (toolName === 'Bash') {
    const command = input.command as string
    return (
      command.includes('rm ') ||
      command.includes('git push') ||
      command.includes('>') // 重定向覆盖文件
    )
  }

  if (toolName === 'Edit') {
    // 文件编辑总是破坏性的(修改现有内容)
    return true
  }

  return false
}

权限检查的逻辑

5-12

主函数 checkPermissions 接收工具名、输入参数、上下文，返回权限决策。

14-25

第一优先级：拒绝规则。如果有明确的拒绝规则，立即返回拒绝，不继续检查。

27-37

第二优先级：允许规则。如果有允许规则且没有拒绝规则，自动允许。

39-49

第三优先级：询问规则。明确标记为"需要询问"的操作。

51-64

默认行为。如果没有匹配的规则，根据权限模式(auto/plan/default)和操作性质(是否破坏性)决定。

67-88

辅助函数：判断操作是否破坏性。例如 rm, git push, 文件编辑都是破坏性的。

💡 核心洞察

权限系统使用优先级链：拒绝 > 允许 > 询问 > 默认。这确保安全规则(拒绝)永远优先于便利性(自动允许)。这是"安全优先"设计原则的体现。
