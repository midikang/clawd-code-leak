设计决策背后的原因

每个架构选择都是为了解决特定问题。让我们看看为什么 Claude Code 这样构建：

🎯 为什么用 Bun 不用 Node.js？

问题：启动速度。开发者打开终端，输入 claude，期待立即响应。

解决：Bun 启动比 Node.js 快 3-4 倍，TypeScript 原生支持，无需编译步骤。

代价：生态系统较小，某些 npm 包可能不兼容。

⚛️ 为什么在终端用 React？

问题：终端 UI 很难写。手动管理光标位置、颜色、状态更新容易出错。

解决：Ink 让你用 React 组件写终端 UI——声明式、可复用、好测试。

好处：团队已经懂 React，复杂 UI（进度条、表单、动画）变简单。

🔧 为什么有 40+ 工具？

问题：编程任务千变万化。读文件、写代码、搜索、运行命令、调试——每个都需要不同能力。

解决：每个工具做一件事，做好。组合使用实现复杂任务。

类比：就像工具箱——有锤子、螺丝刀、扳手，而不是一个"万能工具"。

🛡️ 为什么需要权限系统？

问题：AI 可能会犯错。如果它运行 rm -rf / 怎么办？

解决：每个工具调用都要经过权限检查。破坏性操作必须用户同意。

平衡：自动化 vs 安全。允许模式让流畅操作，询问模式保护关键文件。

🚀 为什么并行预取？

问题：启动时需要读取配置、钥匙串、检查更新——这些操作很慢。

解决：在 main.tsx 加载其他模块的同时，后台启动慢操作。

结果：启动时间从 200ms 降到 135ms——用户感知不到延迟。

🌐 为什么支持 MCP？

问题：不可能内置所有功能。用户想连接 Jira、Notion、自定义数据库。

解决：MCP 是一个协议，让外部服务暴露工具给 AI。

可扩展性：社区可以构建 MCP 服务器，无需修改 Claude Code 核心。

这些知识如何帮助你指导 AI 编程？

学习 Claude Code 的架构不只是了解这个工具——它教会你如何思考软件系统。这些知识直接帮助你：

🎯

做出更好的架构决策

当 AI 问"应该把这个逻辑放在前端还是后端？"，你现在知道：

  • 分层架构的好处（可测试、可替换）
  • 关注点分离（UI vs 业务逻辑 vs 数据层）
  • 什么时候用工具模式（小功能组合成大功能）
🐛

更快找到 bug

当功能不工作时，你知道从哪里查：

  • 数据流问题？查工具调用和 API 往返
  • 权限错误？查权限配置
  • 性能慢？查是否有不必要的串行操作

你能精确告诉 AI："检查 X 组件和 Y 服务之间的数据传递"

⚡

要求性能优化

你学到的模式可以直接应用：

  • "让这两个 API 调用并行执行"
  • "添加懒加载，只在需要时导入这个库"
  • "用特性开关包裹这个实验性功能"
🔒

考虑安全性

权限系统教会你：

  • 哪些操作是破坏性的（删除、覆盖、执行）
  • 如何设计安全检查（白名单 vs 黑名单）
  • 何时需要用户确认

你会告诉 AI："添加确认对话框，因为这会删除数据"

📚

使用正确的术语

你现在可以说：

  • "这个函数应该返回一个 Promise"
  • "用 async/await 处理异步操作"
  • "添加一个 schema 验证输入"
  • "用 feature flag 控制这个功能的开关"

精确的术语让 AI 准确理解你的需求。

🔄

理解 AI 的工作方式

Agent 循环教会你 AI 如何思考：

  • 它不是一次性生成所有代码
  • 它会先探索、再计划、然后执行
  • 它需要反馈循环（工具结果）来调整策略

理解这一点后，你会给 AI 更好的指令："先读取相关文件，然后告诉我你的计划"

最后的挑战

让我们测试一下你是否真正理解了整个系统：

问题 1: 用户输入 "添加登录功能" 后，系统的第一个动作应该是？

问题 5: 你想扩展应用功能，让它能访问 Notion 数据库。受 MCP 设计的启发，最好的方式是？
